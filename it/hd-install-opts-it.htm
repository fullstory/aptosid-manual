Avviare con "fromiso"

Per un uso normale noi raccomandiamo ext3. E' il file system predefinito per sidux ed e' ben mantenuto.

Con questo cheatcode, trucco, potete avviare da una immagine iso su una partizione, (ext2/3), soluzione che e' molto piu' veloce rispetto a quella da CD (mentre l'istallazione su disco fisso con "fromiso" dura una frazione di tempo rispetto sempre al CD).

'fromiso' e' evidentemente molto piu' veloce che da CD/DVD e lascia il lettore libero. Come alternativa potete usare anche QEMU.

Esigenze:

* un grub funzionante (su floppy, installazione su disco fisso, o il Live-CD)
* una immagine iso di sidux, esempio: sidux.iso e un file system linux come ext2/3
* Se cambiate nome alla vostra immagine, ad esempio: da sidux.iso a sidux.ISO dovrete modificare i comandi che seguono secondo i nome che avrete scelto, perche' il sistema di denominazione dei file unix e' sensibile al minuscolo/maiuscolo.

Process:

First read the package manifest of the ISO and find the linux-headers number, it will look something like this:

linux-headers-2.6.24-2.6.24.4.slh.5-common

Therefore the kernel number is:

2.6.24-2.6.24.4.slh.5

Note it very carefully!

Scegliamo un posto per la immagine iso e per i due file che servono, quindi usate nomi corti. Creiamo quindi una cartella base: ad esempio: "/media/sdax" (sda1 o, 2, o 5, o 6, o qualsiasi), create la cartella "sidux" e copiatevi dentro l'immagine iso:

$ sux
mkdir -p /media/sdax/sidux
mv sidux.iso /media/sdax/sidux

ora entriamo in quella cartella e montiamo l'immagine ISO:

mkdir -p /mnt/test
mount -t iso9660 -o loop,ro /media/sdax/sidux/sidux.iso /mnt/test

Copiamo i file vmlinuz e initrd.img dalla iso montata alla cartella sidux where the kernel number is the kernel you acertained from the manifest, and the HD is YOUR known variable::This is an example only:

cp /mnt/test/boot/vmlinuz-2.6.24-2.6.24.4.slh.5  /media/sda5/sidux/
cp /mnt/test/boot/initrd.img-2.6.24-2.6.24.4.slh.5 /media/sda5/sidux/

Ora dovete personalizzare il vostro grub funzionante. Modificate il file /boot/grub/menu.lst e aggiungete le seguenti stringhe DOPO la stringa ### END DEBIAN AUTOMAGIC KERNELS LIST: NOTATE che in una stringa di grub non sono permesse le wildcards (* o ?) per i nomi dei file! The HD is YOUR known variable::This is an example only:

### ISO boot
title sidux 32bit from ISO
kernel (hd0,4)/sidux/vmlinuz-2.6.24-2.6.24.4.slh.5 boot=fll quiet vga=791 fromiso=/sidux/sidux.iso
initrd (hd0,4)/sidux/initrd.img-2.6.24-2.6.24.4.slh.5

Al prossimo riavvio avrete una nuova voce nel menu di grub per avviare l'immagine iso.

fromiso e persist

You can potete avere un sistema live persistente su un disco scrivibile combinando una impostazione fromiso con il cheat-codice 'persist'.

sidux usa aufs per abilitare cio' che e' conosciuto come "copy on write" sopra il cd che permette di scrivere nuovi file e cartelle o aggiornare quelli esistenti sul live-CD, tenendo i nuovi file in memoria. Con il codice persist, una eros salvera' i vostri nuovi file sulla stessa partizione del disco fisso dove avete scritto l'immagine iso del CD che avviate con fromiso.

Usando persist ottente un systema live che esegue tutte le funzionalita' automatiche di un live-CD sidux, inclusa la configurazione automatica dell'hardware ma significa anche che ricreera' gli stessi file ad ogni riavvio..

Ad esempio non salvera' la configurazione di X (scheda video, tastiera e mouse) o il file delle interfacce di rete, cose che rendono questo sistema ideale per l'uso su piu' computer, ma puo' mettervi in difficolta' se dovete poi configurare alcune di queste caratteristiche

Forse l'uso ideale della persistenza e' assieme al nuovo strumento a interfaccia grafica di installazione su usb per creare la vostra personale chiavetta avviabile con i file e i programmi che vi servono abitualmente.

I file saranno salvati in una sottocartella del vostro disco. Questa cartella verra' creata, se ancora non esiste, nel caso che i (cheat)codici fromiso e persist vengano usati assieme.

persist usato sul filesystem FAT, (usato normalmente per installazioni dos/Windows 9x e trovato in via predefinita sui dispositivi a memoria flash come le chiavette usb), necessita della creazione di un grosso, singolo file da usare come loop device che sara' poi da formattare.

Per questo sidux raccomanda sempre l'uso di una partizione ext3 per scriverevi la iso e tutti gli altri file..

Comunque questo file potrebbe essere chiamato sidux-rw e piazzato nella cartella root del vostro dispositivo fat. Supponendo che il disco che volete usare sia montato e scrivibile in /media/sdb1 e vogliate creare un file di 1GB per usarlo come disco persistente, dovreste usare i seguenti comandi:

disc="/media/sdb1"
size=1024
dd if=/dev/zero of=${disc}/sidux-rw bs=1M count=${size} && echo 'y' | /sbin/mkfs.ext3 ${disc}/sidux-rw

Le partizioni NTFS, communemente usate per le installazioni di Windows NT/2000/XP (TM), NON POSSONO essere usate per la persistenza.

Installazione completa di sidux su un dispositivo USB

Per fare una vera installazione di sidux su una chiavetta USB o su un disco fisso USB e facile come una normale installazione su disco fisso. Seguite semplicemente queste linee guida.

Dimensione minima della chiavetta USB:
per una sidux "LITE" sono necessari 1.2giga OLTRE allo spazio per i vostri dati,
per una sidux "FULL" sono necessari 2.1giga OLTRE allo spazio per i vostri dati

Two kinds of installation to USB

1) - fromiso
2) - full (the full installation to a USB stick behaves as normal HD installation and is done through the normal installer).

USB fromiso Installation

The fromiso installation is done through install-usb-gui.bash and is a fromiso install mostly directed to install fromiso on the first partition of an HD + or USB pen.

You have 2 choices:
No Format: the install is made over the first partition with no loss of data.
Format: the disk is erased, all data is lost and single partition created with the chosen file system. Supported file systems are vfat, ext2, ext3.

Data persist is possible and the full partition will be used when ext2 or ext3 are involved. When vfat is used, persistance is made through a file that can be maximum 2GB but not less than 100MB (as it would be of no use).

To start the install-sidux-to-usb go to the submenu System under Kmenu, or use the button that exists on the first screen in th standard HD installer

To create a bootable USB from an installed system you need to have a sidux iso.

Full installation (behaves as normal HD installation)

Iniziamo cme se fosse una installazione normale e scegliamo la partizione dedicata sul dispositivo USB, per esempio "sda1", ma qualsiasi altra partizione sara' valida.

La partizione deve essere formattata con un filesystem linux. Il raccomandato e' ext3. Una partizione esistente verra' riformattata dall'installatore!

Tutti i dati su questa partizione verranno cancellati dalla formattazione!

Ora potrete avviare dal vostro dispositivo USB. Naturalmente dovrete prima abilitare 'Boot from USB' nel BIOS del computer.

Installare sidux con VirtualBox

VirtualBox-ose e' ideale per macchine a 32bit mentre QUEMU lo e' per quella a 64bit.

  • 1. create una immagine disco fisso per VirtualBox
  • 2. avviate la iso con VirtualBox
  • 3. installate l'immagine

Esigenze

RAM raccomandata: 1 giga : Idealmente 512 MB per il guest e 512 MB per lo host. (puo' essere avviato o no, ma non aspettatevi grandi prestazioni).

Spazio sul disco fisso: Mentre VirtualBox in se' e' piuttosto 'magro' (una tipica installazione necesitera' di soli circa 30 MB), le macchine virtuali richiedono invece dei file molto grossi sul disco per rappresentare i propri dischi. Cosi' per installare MS Windows XP (TM), ad esempio, vi servira' un file che puo' crescere facilmente fino a parecchi GB. Per sidux sara' necessario allocare in VirtualBox una immagine di 5 giga oltre ad una locazione di swap.

Fate un dist-upgrade prima di installare i moduli per le macchine virtuali dopo l'installazione di un nuovo kernel o otterrete un 'gcc mismatch, failed module build error' o errore di fallita costruzione del modulo per discrepanza in gcc.(Questo e' valido per ogni modulo esterno)

Il meccanismo:

Installate il pacchetto virtualbox
Premete Applications > System Tools >InnoTek VirtualBox - Virtual Machine
Create una a VM (macchina virtuale), mettete un CD di un sistema operativo nel lettore cd e installatelo. Riavviate.

Installazione

apt-get update
apt-get install virtualbox-ose 
m-a a-i virtualbox-ose

Salvate l'informazione che identifica il corrente sistema nella struttura:

virtualbox-ose-module-$(uname -r)

Dovrete eseguire la seguente stringa:

adduser <username> vboxusers

Usate il wizard di virtualbox per creare una nuova macchina virtuale per sidux. Poi seguite le istruzioni per una normale installazione di sidux.

Installazione automatizzata con svmi

svmi e' avviato dagli script di sidux. Potete lanciare le opzioni manualmente o usare la funzionalita' di installazione automatica, modo non interattivo; leggete opzioni svmi -h per l'autoinstallazione. svmi costruisce il modulo vbox del kernel, lo aggiorna, lo aggiunge in /etc/modules e lo fara' partire automaticamente (altrimenti vbox non verrebbe caricato al riavvio), e aggiorna 'adduser'. Per attivare smvi come root, digitate in una konsole/terminale:

svmi

Installare sidux con QEMU

  • 1. create a immagine di disco fisso per QEMU
  • 2. avviate la iso con QEMU
  • 3. installate nell'immagine

Creare la immagine di disco fisso

Per avviare qemu probabilmente vi serve una immagine di disco fisso. Questo e' un file che immagazzina il contenuto del disco fisso emulato.

Usate il comando seguente per creare il file immagine denominato "sidux.qcow":

qemu-img create -f qcow sidux.qcow 3G

Il parametro "3G" specifica la dimensione del disco; in questo caso appunto 3 GB. Potete usare il suffisso M per megabyte (per esempio "256M"). Non dovreste preoccuparvi troppo riguardo alla dimensione del disco - il formato qcow comprime l'immagine cosicche' lo spazio vuoto non si aggiunge alla dimensione di questo file.

Installare il sistema operativo

Questa e' la prima volta che vi serve l'avvio dell'emulatore. Tenete presente una cosa: quando cliccate dentro la finestra di qemu, il puntatore del mouse viene catturato. Per rilasciarlo premete:

Ctrl+Alt

Se vi serve un floppy avviabile, avviate Qemu con:

qemu -floppy sidux.iso -net nic -net user -m 512 -boot d sidux.qcow

Se il vostro CD-ROM e' avviabile, avviate Qemu con:

qemu -cdrom sidux.iso -net nic -net user -m 512 -boot d sidux.qcow

Ora installate sidux come fareste su un disco fisso reale

Avviare il sistema

Per avviare il sistema, digitate semplicemente:

qemu [hd_image]

Una buoan idea e' utilizzare immagini overlay. In questo modo potete creare immagini di disco fisso una volta e poi dire a Qemu di salvare i cambiamenti in un file esterno. Potete sbarazzarvi del tutto della instabilita' visto che e' cosi' facile ritornare allo stato precedente del sistema.

Per creare una immagine overlay, digitate:

qemu-img create -b [[base''image]] -f qcow [[overlay''image]]

Sostituite la vostra immagine di disco fisso a base_image (nel vostro caso sidux.qcow). Dopo queso potete avviare qemu con:

qemu [overlay_image]

L'immagine originale non verra' toccata. NOTATE: l'immagine di base non puo' essere rinominata o mossa in quanto lo overlay ricorda il completo percorso dell'immagine di base.

Usare qualsiasi partizione reale come partizione primaria singola di una immagine di disco

Talvolta potreste volere usare una delle partizioni del vostro sistema dall'interno di qemu (per esempio se volete avviare sia la macchina reale che qemu usando una certa partizione come root). Potete farlo usando il software RAID in modalita' linere (vi serve il driver del kernel 'linear.ko') e un dispositivo loopback: il trucco e' mettere dinamicamente un master boot record (MBR) all'inizio della partizione reale che volete inglobare in un disco immagine grezza di qemu.

Supponete di avere una partizione pulita e non montata /dev/hdaN con qualche filesystem su di essa che volete che sia parte di una immagine disco di qemu. Prima salvate in un piccolo file il vostro MBR:

dd if=/dev/zero of=/percorso/verso/mbr count=32

Verra' creato un piccolo file di 16 KB (32 * 512 byte). E' importante non crearlo troppo piccolo (anche se a MBR serve solo un signolo blocco di 512 byte), poiche' piu' piccolo sara' e piu' piccola dovra' essere la parte piu' grossa del dispositivo del software RAID, e cio' potrebbe influire sulle prestazioni. Poi impostate un dispositivo di loopback al file MBR:

losetup -f /percorso/verso/mbr

Supponiamo che il dispositivo risultante sia /dev/loop0, in quanto potremmo non avere ancora usato altri loopbacks. Il prossimo passo sara' creare il 'combinato' MBR + /dev/hdaN (immagine disco che usa il sofware RAID:

 modprobe linear
 mdadm --build --verbose /dev/md0 --chunk=16 --level=linear --raid-devices=2 /dev/loop0 /dev/hdaN

Il risultante /dev/md0 sara' quello che userete come immagine grezza di disco qemu (non dimenticate di impostare i permessi in modo che l'emulatore vi possa accedere). L'ultimo (e un po' scaltro) passo e' impostare la configurazione del disco (geometria del disco e tabella della partizioni) in modo che il punto di inizio della partizione primaria nell'MBR si abbini a quello di /dev/hdaN dentro /dev/md0 (un offset di esattamente 16 * 512 = 16384 byte in questo esempio). Fatelo usando fdisk sulla macchina host, non nell'emulatore: la routine predefinita di rilevamento di dischi grezzi produce un 'non kilobyte-roundable offsets' (es. 31.5 KB, come nella sezione precedente) che non puo' essere gestita dal codice del software RAID. Quindi, dall'host:

 fdisk /dev/md0

Li create una partizione primaria singola corrispondente a /dev/hdaN, e giocate con il comando 's'ector dal menu 'x'pert finche' il primo cilindro (dove inizia la prima partizione) si abbini con la dimensione dell'MBR. alla fine 'w'rite (scrivete) il risultato nel file. Avete finito. Ora avete una partizione che potete montare direttamente dal vostro host, come se fosse una parte di una imamgine disco qemu:

 qemu -hdc /dev/md0 [...]

Potete naturalmente impostare in sicurezza qualsiasi bootloader di questa immagine disco che usa qemu, purche' la partizione originale /boot/hdaN contenga gli strumenti necessari.

Usare il modulo acceleratore di QEMU

Gli sviluppatori di qemu hanno creato un modulo opzionale per il kernel per accelerare qemu quasi fino ai livelli nativi. Questo dovrebbe essere caricato con l'opzione

major=0

per automatizzare la creazione del necessario dispositivo /dev/kqemu. Il seguente comando:

echo "options kqemu major=0" >> /etc/modprobe.conf

impostera' modprobe.conf perche' l'opzione venga aggiunta ad ogni caricamente del modulo.

qemu [...] -kernel-kqemu

Il comando sopra abilita la completa virtualizzazione e cosi' migliora considerevolmente la velocita'.

Per attivare qemu:

qemu -cdrom /tmp/pkg/sidux-debug.iso -net nic -net user -m 512

Qui trovate la documentazione ufficiale del Progetto QEMU
