Загрузка с "fromiso"

Для нормального употребления мы рекомендуем файловою систему ext3. Ext3 является стандартной файловой системой для sidux и при этом хорошо поддерживается (maintained).

Этот код обмана стартует ISO файл на жестком диске. Это намного быстрее чем с CD (установки жестких дисков "fromiso" продолжаются только часть нормального времени. Некоторые говорят меньше чем 4 минуты, но 6-8 минуты тоже  быстро).

Это конечно гораздо быстрее, чем с CD, и дисковод стоит в распоряжение. Альтернативно можно использовать также QEMU.

Предпосылки

* функционирующая установка с Grub (на Floppy, установке на жестких дисках или LiveCD)
* ISO-образ sidux.iso и файловая система как ext2/3
* Если ISO-образ переименовывается (например, на sidux.ISO) команды соответствующим образом должны настраиваться, так как система наименования файлов Unix/Linux различает между верхним регистром и нижним регистром.

Сначала мы ищем место для файла ISO Чтобы получить короткие спецификации пути, мы создаем базовый каталог по имени sidux, например, на /dev/hda5, и копируем ISO внутрь:

# mkdir /media/hda5/sidux
# mv sidux.iso /media/hda5/sidux

Мы подсоединяем ISO:

# mkdir -p /mnt/test
# mount -t iso9660 -o loop,ro /media/hda5/sidux/sidux.iso /mnt/test

Мы копируем файлы miniroot.gz и vmlinuz из подсоединённого ISO:

# cp /mnt/test/boot/vmlinuz /media/hda5/sidux/
# cp /mnt/test/boot/miniroot.gz /media/hda5/sidux/

Теперь GRUB должен еще настраиваться Для этого мы открываем /boot/grub/menu.lst и добавляем следующее:

### ISO boot
title sidux 32bit from ISO
kernel (hd0,4)/sidux/vmlinuz boot=fll quiet vga=791 fromiso=/sidux/sidux.iso
initrd (hd0,4)/sidux/miniroot.gz

При следующей загрузке новый выбор появляется в меню Grub: "sidux 32bit fromISO".

Установка sidux на USB-Stick или на USB жесткий диск

Установка  sidux на USB-Stick или USB-жесткий диск является теперь так же проста как нормальная установка жестких дисков. Следуйте просто за этим коротким руководством.

Рекомендованная минимальная величина для установки на USB-Stick является:

sidux "Lite" нуждается в 1.2 ГБ. К этому добавляется еще место для данных.
sidux "FULL" нуждается в 2.1 ГБ. К этому добавляется еще место для данных.

Поэтому рекомендован для установки Lite на USB-Stick с минимум 2 ГБ, для установки Full на USB-Stick с минимум 3 ГБ.

Теперь установка USB является частью sidux-installers и может им проводятся.

Начните нормальную установку и выберите раздел на диске USB, на который должен устанавливаться sidux - мы говорим в последующем о "sda1", но это же самое имеет значение для каждого другого раздела жесткого диска USB.

Grub должен устанавливаться в разделе установки (а не в MBR)

Раздел должен использовать файловою систему Linux. Рекомендована ext3 файловая система. Уже существующий раздел форматирует sidux-installer заново.

Все данные на этом разделе при этом удаляются!

Теперь Вы можете, перезагрузится с Вашей USB установки. Активируйте раньше в BIOS "Boot from USB".

Установка sidux с QEMU

Создание имиджа жестких дисков (iso)

Чтобы работал qemu, он нуждается в имидже жестких дисков. Это файл, который несет содержание эмулируемого жесткого диска.

Используют команду:

qemu-img create -f qcow sidux.qcow 3G

С этим создают файл имиджа с именем "sidux.qcow". Параметр "3G" специфицирует величину диска, в этом случае 3 ГБ. Суффикс М используется для мегабайта (например "256M"). Не надо слишком задумываться о величине на диске - формат qcow архивирует имидж, так что неиспользованное место не прибавляется себя к величине файла.

Инсталляция операционной системы

Теперь эмулятор запускается первый раз. При этом нужно обратить внимание: Eсли в пределах окна qemu нажимают при помощи мыши, станет её указатель "пойманным". Чтобы освободить его снова, нажимают на Ctrl+Alt.

Если нуждаются в самозагружаемой дискете, qemu запускается таким образом:

qemu -floppy sidux.iso -net nic -net user -m 512 -boot d sidux.qcow

Если СD-ROM самозагружаем, qemu запускается таким образом:

qemu -cdrom sidux.iso -net nic -net user -m 512 -boot d sidux.qcow

Теперь sidux может инсталлироваться как на реальный жесткий диск.

Применение системы

Чтобы запускать систему, вводят:

qemu [hd_image]

Хорошей идеей является применение т.н. "напластованных имиджей " (overlay images). Таким образом, имидж должен создаваться только один раз и qemu сохраняет изменения вне этого имиджа. Cистема является таким образом более стабильной, так как очень просто, на предыдущее инсталляционное состояние возвращаться.

Чтобы создавать "напластованный имидж" (overlay image), вводят:

qemu-img create -b [[base''image]] -f qcow [[overlay''image]]

Теперь имидж жестких дисков заменяет базовый имидж (base_image), в нашем случае win.qcow). Затем может запускаться qemu:

qemu [overlay_image]

Оригинальный имидж остается неприкосновенным. Однако вы должны обращать внимание на то, что это базовый имидж не может ни переименовываться, ни перемещаться, так как оверлей нуждается в точном, абсолютном пути базового имиджа.

Как каждый настоящий раздел может использоваться как первоначальный раздел имиджа жестких дисков.

Иногда хотелось бы использовать системный раздел из qemu (хотелось бы например, как настоящую инсталляцию, так и виртуальную инсталляцию qemu запускать с данного раздела как root). Это происходит посредством Software-RAID в линейном модусе (для этого нуждаются в модуле ядра linear.ko) и устройство Loopback: При этом фокус состоит в том, что Master Boot Record (MBR) ставят динамично в начале реального раздела, который должен встраиваться в qemu имидж Raw Disk.

Предположим, имеют простой, не включенный раздел /dev/hdaN с файловой системой, который должен стать частью имиджа жестких дисков qemu. Как первое создают маленький файл, который должен содержать MBR:

dd if=/dev/zero of=/path/to/mbr count=32

С этим создаётся файл размером 16 KB (32 * 512 bytes). Важ­но, не калькулировать этот файл слишком маленьким (даже если MBR нуждается только в одном блоке размером 512 байтов), так как Chunk-Size программного RAID устройства должно быть тем меньше, чем меньше тот файл является, что снова будет иметь влияние на производительность. Затем может устройство Loopback присоединяться к файлу MBR:

losetup -f /path/to/mbr

Если мы исходим из того, что устройство /dev/loop0 называется, так как еще никакой следующий Loopback не существует. Следующий шаг - это составление соединенного "merged" имиджа с MBR +/dev/hdaN при использовании Software-RAIDS:

 modprobe linear
 mdadm --build --verbose /dev/md0 --chunk=16 --level=linear --raid-devices=2 /dev/loop0 /dev/hdaN

Следующий из этого /dev/md0 используется как qemu-Raw-Diskimage (постановку прав не забыть, вместе с тем чтобы эмулятор мог его использовать). Последний (и несколько щекотливый) шаг - это производство конфигурации жестких дисков (геометрия диска и таблицы разделов), так что исходная точка первоначального раздела в MBR совпадает с одним из /dev/hdaN в пределах/dev/md0 (смещение точно 16 * 512 = 16 384 Байт в этом примере). Для этого используется fdisk на машине Host, а не на эмуляторе: В основных установках следует узнавание Raw Disk qemu часто в комплектах Off, которые не могут быть округленны в килобайты (как 31.5 KБ в предшествовавшем разделе), что снова кодом Software-RAIDs не может пониматься. Поэтому вводят на компьютор хоста:

 fdisk /dev/md0

Там создают первоначальный раздел соответствующим образом /dev/hdaN и играют командой 's' (Sector) в меню 'x' (эксперт), до тех пор, пока первый цилиндр (где начинается первый раздел) точно подходит к величине MBR. Наконец, с 'w' (write) результат записывают в файл: вместе с этим работа завершена. Теперь имеется в наличии раздел, который непосредственно может включаться как машиной Host так и имиджем жестких дисков qemu:

 qemu -hdc /dev/md0 [...]

Конечно, можно безопасно записать каждый загрузчик на имидж жестких дисков с qemu, предполагается, оригинальный раздел /boot/hdaN содержит для этого необходимые приложения.

Модуль QEMU Accelerator

Разработчики qemu создали опциональный модуль ядра, чтобы qemu мог набирать скорость почти до исконного уровня. Этот модуль загружаются с этой опцией: Этот модуль загружаются с этой опцией:

major=0

вместе с этим автоматизируется установка необходимого устройства /dev/kqemu. Следующие команды являются:

echo "options kqemu major=0" >> /etc/modprobe.conf

Она расширяет modprobe.conf, чтобы обеспечивать, то чтобы опция модуля активируется при каждой загрузке модуля.

qemu [...] -kernel-kqemu

Вместе с этим делается возможным достичь полную виртуализацию и значительно повысить скорость.

Активизация qemu:

qemu -cdrom /tmp/pkg/sidux-debug.iso -net nic -net user -m 512
